## 1. 서비스 개요

### 배경

확대정부 기조에 따라 정부지원사업 공고가 대폭 증가하고 있다. 그러나 인력이 부족한 중소기업에서는 사업 신청에 필요한 부대서류를 담당자 1인이 혼자 챙겨야 하는 구조다.

**현재 중소기업 담당자의 실상:**
- 서류별 유효기간을 놓쳐 사업 신청이 반려되는 케이스 빈번
- 서명·도장 이미지 파일을 수기로 찾아 크기 맞춰 붙여넣는 반복 작업
- 동의서·서약서를 프린트 → 손서명 → 스캔하는 아날로그 프로세스
- NDA·계약서 체결 시 왕복 2회 프린트·스캔·전달
- 프리랜서·외주 계약 때마다 매번 동일 서류 수집 반복
- 담당자가 바뀌면 인수인계 거의 안 됨

### 서비스 한 줄 정의

> 정부지원사업 부대서류 유효기간 검증부터 서류 작성·서명 자동화, 전자계약 체결까지 — 중소기업 특화 서류·계약 관리 플랫폼

### 경쟁 서비스 비교

| 서비스 | 포지션 | 한계 |
|---|---|---|
| 모두싸인 | 범용 전자계약 | 정부서류 비특화. 상대방 회원가입 강제 유도. 중소기업 실무 맥락 없음 |
| DocuSign | 글로벌 전자계약 | B2B 고가. 한국 공공서류 완전 비특화 |
| **ẞDM** | 정부지원사업 특화 원스톱 서류·계약 플랫폼 | - |

**핵심 차별점**: 상대방 회원가입 없이 인증서 or 간편인증 하나로 계약 체결 완료

## 2. 타겟 사용자

- **1차**: 정부지원사업을 상시 신청하는 중소기업·스타트업 경영지원 담당자
- **2차**: 여러 기업 서류를 대행하는 컨설팅사·법인

**UX 원칙**: IT 친숙도가 낮은 담당자도 쓸 수 있어야 함. 결과만 보여줄 것. 설정 최소화.

## 4. 초기 배포 전략

## 6. Phase 1 — 부대서류 유효기간 자동 검증

### 작동 방식

1. 지정 폴더(데스크탑) 또는 업로드(웹앱)에 서류 파일 저장
2. 파일 자동 감지 → 텍스트 추출 또는 OCR 처리
3. 서류 헤더 기반 문서 종류 자동 분류
4. 문서 종류별 룰셋으로 유효기간 추출
5. 현재 날짜와 비교 → 결과 대시보드 표시

### 결과 표시 기준

| 상태 | 기준 |
|---|---|
| 🟢 유효 | 만료까지 30일 이상 |
| 🟡 만료 임박 | 만료까지 30일 이내 |
| 🔴 만료 | 유효기간 초과 |
| ⚪ 확인 필요 | OCR 실패 또는 날짜 인식 불가 |

### 대상 서류 목록

| 서류명 | 유효기간 | 비고 |
|---|---|---|
| 사업자등록증 | 없음 (변경 시 재발급) | 최신본 여부 확인 |
| 중소기업확인서 | 매년 3월 31일까지 | 발급일 무관, 연 단위 고정 만료 |
| 법인등기부등본 | 발급일로부터 3개월 | 공고마다 상이하나 통상 3개월 |
| 4대보험완납증명서 | 발급일로부터 30~90일 | 공고마다 상이 |
| 납세증명서 (국세완납증명) | 발급일로부터 30일 | 유효기간 짧아 실무에서 가장 자주 문제 발생 |
| 지방세납세증명서 | 발급일로부터 30일 | 위텍스 발급 |
| 벤처기업확인서 | 확인일로부터 3년 | 해당 기업만 |
| 이노비즈(Inno-Biz) 확인서 | 3년 | 해당 기업만 |
| 메인비즈(Main-Biz) 확인서 | 3년 | 해당 기업만 |
| 기업부설연구소 인정서 | 별도 만료일 명시 | R&D 사업 필수 |
| 고용보험 사업장 가입자 명부 | 발급일 기준 최신본 | 고용 관련 사업 |
| 건강보험료 납부확인서 | 발급일로부터 30~90일 | 인력 관련 사업 |

**보류**: 재무제표 — 결산 기준년도·감사보고서 여부 등 변수가 많아 추후 검토

## 7. Phase 2 — 서류 작성·서명 자동화

### 현재 실무 vs 개선

| 현재 | 개선 후 |
|---|---|
| 프린트 → 손서명/도장 → 스캔 → 업로드 | PDF 열고 드래그 앤 드롭 → 다운로드 |
| 서명 이미지 파일 찾고 크기 맞추고 붙여넣기 반복 | 왼쪽 패널 담당자 카드 → 원하는 위치에 드롭 |
| 사업마다 담당자 정보 수기 입력 | 세트 선택 한 번으로 전체 세팅 완료 |

### 담당자 프로필 세트

**담당자별 등록 정보:**
- 이름, 직책, 소속
- 서명 이미지
- 도장 이미지
- 자주 쓰는 텍스트 (사업자번호, 회사명, 주소, 연락처 등)

**세트 기능:**
- 여러 담당자를 묶어 세트로 구성
- 사업 신청 시 세트 선택 한 번으로 전체 세팅 완료

**PDF 작성 UI:**
- PDF 열면 왼쪽 패널에 담당자 카드 나열
- 카드를 PDF 원하는 위치로 드래그 앤 드롭
- 서명·도장·텍스트 자동 삽입
- 완성 PDF 다운로드

**대상 서류:**
- 보안서약서
- 개인정보수집·활용·제공 동의서
- AI 개인정보보호 동의서
- 기타 기관별 제출 양식

**양식 DB 구축 방식 (미결):**
- A안: 주요 기관 양식 사전 등록 → 자동 위치 지정. 완성도 높으나 운영 부담
- B안: 사용자가 직접 드래그로 위치 지정 → 운영 부담 없으나 UX 다소 복잡. 단, 현재 프린트·스캔 대비 압도적으로 편함

## 8. Phase 3 — 전자계약 체결

### 3-A: 기업 간 계약

**대상 문서**: NDA, MOU, 협약서, 계약서

| 현재 | 개선 후 |
|---|---|
| 왕복 2회 프린트·서명·스캔·전달 | 링크 클릭 → 인증 → 서명 등록 → 완료 |

**갑(회원사)**: 표준 양식 입력 → 임시링크 + DRM 생성 → 전달

**을(상대방)**: 회원가입 없음. 링크 접속 → 인증 → 서명·도장·연락처 등록 → 완료

### 3-B: 프리랜서·외주·용역 계약

- 프리랜서가 한 번 서류 등록 → 링크로 공유 → 갑이 클릭해서 바로 수령
- **서류 지갑**: 을도 플랫폼 유저로 유입되는 구조

### 인증 트랙

| 구분 | 인증 방식 |
|---|---|
| 법인 | 법인 공동인증서 |
| 개인사업자 | 카카오페이서명·네이버·토스·PASS 간편인증 |
| 개인 (프리랜서·용역) | 휴대폰 본인인증 + 간편인증 |

## 9. 개발 스택

| 항목 | 선택 | 비고 |
|---|---|---|
| Frontend | React | - |
| Backend | FastAPI (Python) | - |
| Desktop | Electron | Windows 중심, 8GB RAM 이상 환경 |
| OCR | PaddleOCR | 한국어 정확도 우수 |
| OCR 고도화 | PaddleOCR 파인튜닝 | 실사용 데이터 100건↑ 쌓인 후 시작 |
| PDF 파싱 | PyMuPDF | 텍스트 추출 우선, 실패 시 OCR 전환 |
| PDF 보조 | pdfplumber | 표 구조 처리 |
| File Watcher | watchdog | 중복 실행 방지 로직 포함 |
| DB | Supabase (PostgreSQL) | 추후 자체 NAS PostgreSQL 마이그레이션 |
| 배포 | 추후 결정 | - |

### OCR 처리 플로우

```
PDF 입력
  ↓
PyMuPDF 텍스트 추출 시도
  ↓ 성공          ↓ 실패 (이미지 PDF)
텍스트 사용    PaddleOCR 처리
  ↓
헤더 기반 문서 종류 판별
  ↓
문서별 룰셋으로 날짜 추출
  ↓
유효기간 계산 → 결과 표시
```

### 초기 데이터 전략

- MVP: 기본 PaddleOCR + 룰 기반 후처리로 출시
- 보유 서류 샘플 → 학습이 아닌 **테스트 데이터**로 활용 (정확도 검증용)
- 배포 후 실사용 데이터 100건↑ 확보 시점에 파인튜닝 시작
- 반복 학습으로 정확도 고도화 → 기술 진입장벽 형성
